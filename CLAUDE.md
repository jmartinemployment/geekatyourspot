# content-writer-v2 has no database

content-writer-v2 (separate repo, `~/development/content-writer-v2`) has no database — its only durable output is `.html` files committed to this repo's `content-writer-output/` directory via GitHub's Git Data API. This repo never talks to content-writer-v2's API, and there is no database anywhere in this pipeline to query or reconcile against.

**No reading/rendering pipeline currently exists in this repo.** All code built against content-writer-v2's old export shape has been deleted: `src/lib/content-writer/content.ts`, `src/lib/content-writer/mdxFlatParser.ts`, `src/components/content-writer/content-page.tsx`, `src/components/content-writer/flat-mdx-viewer.tsx`, the `src/app/{blog,tools,use-cases}/[department]/[slug]/page.tsx` dynamic routes, and the `src/app/flat-mdx-demo/*` prototype pages. That shape turned out to be mislabeled (files were GFM Markdown text despite the `.html` extension and "HTML" field/type names, never real HTML). Every Markdown-parsing dependency was removed along with it — `react-markdown`, `remark-gfm`, `remark-mdx`, `gray-matter`, `@mdx-js/*`, `@next/mdx` are all gone from `package.json`. Do not reintroduce a Markdown-parsing library or any of the above files here.

content-writer-v2 is being rewritten (in progress) to export a standalone `<!doctype html>` file per page — real HTML, no frontmatter, built from structured content server-side via AngleSharp. Per that design, this repo's responsibility starts only once a valid file exists at the expected path in `content-writer-output/`; how it gets served (static asset vs. some other approach) is not yet decided or built here — don't assume the old dynamic `[department]/[slug]` + content.ts pattern is the plan going forward.
